# What is SonarQube?

SonarQube is:

# a Static Code Analysis Tool

Used for:

* code quality analysis
* bug detection
* vulnerability detection
* maintainability checks

---

# MOST IMPORTANT EXAM DEFINITION

SonarQube:

> analyzes source code without executing it.

This is:

# Static Analysis

VERY HIGH probability MCQ.

---

# PART 3 — Key SonarQube Concepts

EXTREMELY IMPORTANT.

---

# 1. Bugs

Potential coding errors.

Example:

* null pointer issue
* wrong logic

---

# 2. Vulnerabilities

Security issues.

Example:

* SQL injection
* hardcoded passwords

---

# 3. Code Smells

Bad coding practices.

Example:

* duplicate code
* huge functions
* unreadable logic

VERY HIGH probability MCQ.

---

# 4. Technical Debt

Future maintenance cost caused by poor code.

---

# 5. Quality Gate (VERY IMPORTANT)

A set of rules deciding:

# PASS or FAIL

Example:

* no critical vulnerabilities
* code coverage > 80%

---

# MOST IMPORTANT CONCEPT

Quality Gate determines:

> whether pipeline should continue.

VERY important in CI/CD.

---

# PART 7 — SonarQube + CI/CD Integration

VERY HIGH probability.

---

# GitHub Actions Integration

```yaml id="sq6"

  run: sonar-scanner
```

---

# Full Example

```yaml id="sq7"
name: CI Pipeline

on:
  push:
    branches:
      - main

jobs:
  scan:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: SonarQube Scan
        run: sonar-scanner
```

VERY important practical pattern.

# PART 11 — What is OPA?

OPA =

# Open Policy Agent

Used for:

* policy enforcement
* access control
* security rules
* compliance automation

---

# MOST IMPORTANT DEFINITION

OPA:

> Policy-as-Code engine.

VERY HIGH probability MCQ.

---

# PART 12 — What is Policy-as-Code?

Instead of manually checking security rules:

* rules are written as code.

Example:

* who can access resource
* deployment restrictions
* container security policies

---

# REAL EXAMPLES

OPA can enforce:

```text id="opa1"
Only admins can deploy
Only approved Docker images allowed
No privileged containers
```

---

# PART 13 — OPA Workflow

---

# Full Flow

```text id="opa2"
Request/Input
      ↓
OPA Evaluates Policies
      ↓
ALLOW / DENY Decision
```

---

# PART 14 — Rego Language (EXTREMELY IMPORTANT)

OPA policies are written in:

# Rego

VERY HIGH probability MCQ.

---

# Example Rego Policy

```rego id="opa3"
package example

default allow = false

allow {
  input.user == "admin"
}
```

---

# Understand This Carefully

---

# package

```rego id="opa4"
package example
```

Policy namespace.

---

# default

```rego id="opa5"
default allow = false
```

Default deny.

VERY important security principle.

---

# allow rule

```rego id="opa6"
allow {
  input.user == "admin"
}
```

If user is admin:

* allow access

Else:

* deny

---

# PART 15 — Another OPA Example

VERY useful for understanding.

---

# Only HTTPS Allowed

```rego id="opa7"
package security

default allow = false

allow {
  input.protocol == "https"
}
```
